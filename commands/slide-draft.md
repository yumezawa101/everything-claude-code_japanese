# Slide Draft - スライド構成ドラフト作成

プレゼン資料のスライド構成を Markdown 形式で出力する。
パワポに直接出力せず、構成とテキストの素材を作ることに集中する。

## Phase 1: ヒアリング（必須・スキップ不可）

ドラフト作成前に、以下を必ず質問して要件を明確にする。
ユーザーが曖昧な場合は具体例を出して選ばせる。

### 質問1: 目的と対象

```
この資料の目的は何ですか？
  a) 承認を得る（企画書、提案書）
  b) 報告する（進捗報告、結果報告）
  c) 説明する（技術説明、業務フロー）
  d) 教育する（研修資料、ガイド）

誰に見せますか？
  a) 経営層 → 結論ファースト、数字重視
  b) クライアント → 課題共感、ベネフィット重視
  c) チームメンバー → 具体的な手順、アクション重視
  d) 不特定多数 → シンプル、自己完結型
```

### 質問2: 伝えたいこと

```
この資料を見た相手に、最終的にどうなってほしいですか？
（例: 「予算を承認してほしい」「この設計方針でOKをもらいたい」）

伝えたい核心メッセージを1文で言うと？
（例: 「筋トレアプリは月額課金で半年で黒字化できる」）
```

### 質問3: 制約

```
- スライド枚数の目安は？（5枚 / 10枚 / 20枚 / 制限なし）
- 発表時間は？（5分 / 15分 / 30分 / 読み物として配布）
- 使いたいデータや図表はすでにある？
- パワポテンプレートはある？（あれば共有してください）
```

### 質問4: アウトプットイメージの確認

```
イメージに近いのはどれですか？
  a) テキスト中心（箇条書きメイン）
  b) ビジュアル重視（図・グラフ・画像が多い）
  c) データ重視（表・統計・比較が多い）
  d) ストーリー型（起承転結、読ませる構成）
```

## Phase 2: 構成設計

ヒアリング結果に基づいてスライド構成を設計する。
presentation-patterns スキルのフレームワークを参照。

目的別の推奨構成：

### 承認を得る（企画書・提案書）
```
1. タイトル
2. エグゼクティブサマリー（結論を先に）
3. 課題/背景
4. 提案内容
5. 期待効果（定量）
6. 実施計画/スケジュール
7. コスト/リソース
8. リスクと対策
9. 次のステップ
```

### 報告する（進捗・結果）
```
1. タイトル
2. サマリー（結論）
3. 前回からの進捗
4. 成果/数字
5. 課題/未解決事項
6. 次のアクション
```

### 説明する（技術・業務フロー）
```
1. タイトル
2. 全体像/概要図
3. 詳細説明（セクション分割）
4. 具体例/デモ
5. FAQ/補足
```

## Phase 3: Markdown ドラフト生成

以下のフォーマットで各スライドを出力する：

```markdown
---
# スライド1: [タイトル]

## 見出し
本文テキスト

- 箇条書き1
- 箇条書き2

**スピーカーノート**: 口頭で補足する内容

**ビジュアル案**: [このスライドに入れたい図表の説明]
→ /diagram-prompt で画像生成プロンプトを作成可能

---
```

## 注意事項

- 1スライド = 1メッセージの原則を守る
- テキストは箇条書き3-5項目を目安に
- 数字は大きく見せる（「売上30%増」をスライドの主役にする）
- ビジュアル案はテキストで記述し、実際の画像生成は /diagram-prompt に委ねる
- パワポテンプレートが共有された場合は presentation-patterns スキルで構造を読み取る
