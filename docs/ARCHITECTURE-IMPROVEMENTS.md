# アーキテクチャ改善の推奨事項

本ドキュメントは、Everything Claude Code (ECC) プロジェクトのアーキテクトレベルの改善点を記録します。保守性、一貫性、長期的な品質の向上を目指す Claude Code コーディングアーキテクトの視点で作成されています。

---

## 1. ドキュメントと信頼できる唯一の情報源

### 1.1 Agent / コマンド / スキル数の同期

**問題:** AGENTS.md は「13の専門 agent、50以上のスキル、33のコマンド」と記載していますが、リポジトリには実際には **16の agent**、**65以上のスキル**、**40のコマンド** があります。README やその他のドキュメントも数値が異なっています。これはコントリビューターやユーザーに混乱を引き起こします。

**推奨:**

- **信頼できる唯一の情報源:** 数値（およびオプションでテーブル）をファイルシステムまたは小さなマニフェストから導出する。選択肢:
  - **オプション A:** ファイルシステムの `agents/*.md`、`commands/*.md`、`skills/*/SKILL.md` をスキャンして JSON/Markdown を出力するスクリプト（例: `scripts/ci/catalog.js`）を追加。CI とドキュメントがこれを参照。
  - **オプション B:** agent、コマンド、スキルをメタデータ付きでリストする `docs/catalog.json`（または YAML）を1つ管理。スクリプトとドキュメントがここから読み取る。追加/削除時の更新規律が必要。
- **短期対応:** AGENTS.md、README.md、CLAUDE.md を実際の数値に手動で同期し、新しい agent（例: chief-of-staff、loop-operator、harness-optimizer）を agent テーブルに追加。

**影響度:** 高 -- 第一印象とコントリビューターの信頼に影響。

---

### 1.2 コマンド → Agent / スキル マップ

**問題:** 「どのコマンドがどの agent やスキルを使用するか」を示す、機械可読または人間可読の単一マップが存在しない。この情報は README のテーブルや個々のコマンド `.md` ファイルに散在しており、ドリフトする可能性がある。

**推奨:**

- 各コマンドの名前、説明、主要 agent、参照スキルをリストする**コマンドレジストリ**（例: `docs/` 内、またはコマンドファイルの frontmatter として）を追加。コマンドファイルの内容から生成するか、手動で管理可能。
- ドキュメント（例: `docs/COMMAND-AGENT-MAP.md`）または生成カタログに「マップ」を公開し、発見可能性とツーリング（例: 「tdd-guide を使用するコマンドはどれか?」）に活用。

**影響度:** 中 -- 発見可能性とリファクタリングの安全性が向上。

---

## 2. テストと品質

### 2.1 テスト探索 vs ハードコードリスト

**問題:** `tests/run-all.js` はテストファイルの**ハードコードリスト**を使用している。新しいテストファイルは誰かが `run-all.js` を更新しない限り実行されないため、カバレッジが不完全になる可能性がある。

**推奨:**

- **Glob ベースの探索:** パターン（例: `tests/` 配下の `**/*.test.js`）でテストファイルを探索し実行。特殊なケース用にオプションの許可/拒否リストを設ける。これにより新しいテストは自動的にスイートの一部になる。
- 探索されたテストを実行し結果を集約する単一のエントリポイント（`tests/run-all.js`）を維持。

**影響度:** 高 -- 新しいテストが存在するが実行されないというリグレッションを防止。

---

### 2.2 テストカバレッジメトリクス

**問題:** カバレッジツール（例: nyc/c8/istanbul）が存在しない。プロジェクト自身のスクリプトに対して「80%以上のカバレッジ」を主張できない。カバレッジは暗黙的。

**推奨:**

- Node スクリプト用のカバレッジツール（例: `c8` または `nyc`）を導入し、CI で実行。ベースライン（例: 60%）から始めて徐々に引き上げるか、少なくとも失敗させずに CI でカバレッジを報告してトレンドを可視化。
- `scripts/`（lib + hooks + ci）を主要ターゲットとし、必要に応じて単発スクリプトを除外。

**影響度:** 中 -- プロジェクト自身の AGENTS.md ガイダンス（80%以上のカバレッジ）に整合し、未テストパスを可視化。

---

## 3. スキーマとバリデーション

### 3.1 CI で Hooks JSON スキーマを使用

**問題:** `schemas/hooks.schema.json` が存在し hook 設定の形状を定義しているが、`scripts/ci/validate-hooks.js` はこれを**使用していない**。バリデーションが重複（VALID_EVENTS、構造）しており、スキーマからドリフトする可能性がある。

**推奨:**

- `validate-hooks.js` で JSON スキーマバリデーター（例: `ajv`）を使用し、`hooks/hooks.json` を `schemas/hooks.schema.json` に対してバリデート。バリデーターを構造の信頼できる唯一の情報源とし、hook 固有のチェック（例: インライン JS 構文）のみをスクリプトに残す。
- スキーマとバリデーターの同期を確保し、hooks.json の `$schema` を通じた IDE/エディターバリデーションを可能にする。

**影響度:** 中 -- ドリフトを低減し、hook 編集時のコントリビューター体験を改善。

---

## 4. クロスハーネスと i18n

### 4.1 スキル/Agent サブセットの同期 (.agents/skills, .cursor/skills)

**問題:** `.agents/skills/`（Codex）と `.cursor/skills/` は `skills/` のサブセット。メインリポジトリでスキルを追加・削除する場合、これらのサブセットを手動で更新する必要があり、忘れられる可能性がある。

**推奨:**

- CONTRIBUTING.md にスキル追加時に `.agents/skills` と `.cursor/skills` の更新が必要な場合があること（およびその方法）を文書化。
- オプション: `skills/` とサブセットを比較し、規約やマニフェストに基づいて一方にあって他方にないスキルがある場合に失敗または警告する CI チェックまたはスクリプト。

**影響度:** 低～中 -- クロスハーネスのドリフトを低減。

---

### 4.2 翻訳のドリフト (docs/ zh-CN, zh-TW, ja-JP)

**問題:** `docs/` の翻訳は agent、コマンド、スキルを複製している。英語ソースの進化に伴い、翻訳は明確なプロセスやツーリングなしに古くなる可能性がある。

**推奨:**

- **翻訳プロセス**を文書化: 更新タイミング（例: リリース時）、各ロケールの担当者、古いコンテンツの検出方法（例: ファイルリストや主要セクションの差分）。
- 検討: 翻訳ステータスファイル（例: `docs/i18n-status.md`）または翻訳ファイルの存在/タイムスタンプをチェックし、英語が翻訳より最近更新された場合に警告する CI。
- 長期: 翻訳が英語ソースと同じ構造を参照するよう、抽出/プレースホルダー形式（例: i18n キー）を検討。

**影響度:** 中 -- 英語以外のユーザーの体験を改善し、古い翻訳による混乱を低減。

---

## 5. Hooks とスクリプト

### 5.1 Hook ランタイムの一貫性

**問題:** ほとんどの hook は `run-with-flags.js` 経由で Node スクリプトを呼び出すが、一部のパスは `run-with-flags-shell.sh` + `observe.sh` を使用。混在するランタイムは文書化されているが、時間をかけて簡素化できる可能性がある。

**推奨:**

- 可能な場合、新しい hook には Node を優先（クロスプラットフォーム、単一ランタイム）。シェルが必要な場合はその理由を文書化し、表面積を小さく保つ。
- `ECC_HOOK_PROFILE` と `ECC_DISABLED_HOOKS` がすべてのコードパス（シェル含む）で尊重されることを確認し、動作の一貫性を保つ。

**影響度:** 低 -- 現在の設計を維持しつつ、より多くの hook が Node に移行すれば改善。

## 7. クイックウィン（即座に対応可能）

1. **AGENTS.md を更新:** agent 数を16に設定。chief-of-staff、loop-operator、harness-optimizer を agent テーブルに追加。スキル/コマンド数をリポジトリに合わせる。
2. **テスト探索:** `run-all.js` を `tests/` 配下の `**/*.test.js` を探索するよう変更（オプションの許可リスト付き）し、新しいテストが常に実行されるようにする。
3. **Hook スキーマの接続:** `validate-hooks.js` で ajv（または類似ツール）を使用して `hooks/hooks.json` を `schemas/hooks.schema.json` に対してバリデートし、hook 固有のチェックのみをスクリプトに残す。

これら3つは1～2回のセッションで完了でき、一貫性と信頼性を大幅に改善する。
