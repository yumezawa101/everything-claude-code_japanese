# Context Budget

Claude Codeセッションで読み込まれたすべてのコンポーネントのトークンオーバーヘッドを分析し、コンテキスト空間を取り戻すための実行可能な最適化を提示する。

## 使用タイミング

- セッションパフォーマンスが遅く感じる、または出力品質が低下している
- 最近多くのスキル、エージェント、MCPサーバーを追加した
- 実際にどれだけのコンテキスト余裕があるか知りたい
- コンポーネントを追加予定で余裕があるか確認が必要
- `/context-budget`コマンドの実行時（このスキルがバック）

## 仕組み

### フェーズ1：インベントリ

すべてのコンポーネントディレクトリをスキャンし、トークン消費を推定する。

### フェーズ2：分類

すべてのコンポーネントをバケットに分類：

| バケット | 基準 | アクション |
|--------|----------|--------|
| **常に必要** | CLAUDE.mdで参照、アクティブなコマンドをバック、現在のプロジェクトタイプに一致 | 保持 |
| **時々必要** | ドメイン固有、CLAUDE.mdで参照なし | オンデマンド有効化を検討 |
| **ほとんど不要** | コマンド参照なし、内容重複、明確なプロジェクト一致なし | 削除またはレイジーロード |

### フェーズ3：問題検出

以下の問題パターンを特定：

- **肥大化したエージェント説明** -- frontmatterの30語超の説明は全Taskツール呼び出しで読み込まれる
- **重いエージェント** -- 200行超のファイルは各スポーン時にTaskツールコンテキストを膨張させる
- **冗長なコンポーネント** -- エージェントロジックを重複するスキル、CLAUDE.mdを重複するルール
- **MCPのオーバーサブスクリプション** -- 10以上のサーバー、無料で利用可能なCLIツールをラップするサーバー
- **CLAUDE.mdの肥大化** -- 冗長な説明、古いセクション、ルールにすべき指示

### フェーズ4：レポート

コンテキストバジェットレポートを生成。コンポーネント別の内訳、発見された問題、上位3つの最適化、潜在的な節約を含む。

## ベストプラクティス

- **トークン推定**: 散文は`単語数 x 1.3`、コード重視ファイルは`文字数 / 4`を使用
- **MCPが最大のレバー**: 各ツールスキーマに約500トークン。30ツールのサーバーはすべてのスキルの合計よりコストが高い
- **エージェント説明は常に読み込まれる**: エージェントが呼び出されなくても、description フィールドはすべてのTaskツールコンテキストに存在
- **変更後に監査**: エージェント、スキル、MCPサーバーの追加後に実行してクリープを早期に捕捉
