# コードベースオンボーディング

未知のコードベースを体系的に分析し、構造化されたオンボーディングガイドを生成する。新しいプロジェクトに参加する開発者や、既存リポジトリで初めてClaude Codeをセットアップする場合向け。

## 使用タイミング

- Claude Codeでプロジェクトを初めて開く場合
- 新しいチームやリポジトリに参加する場合
- ユーザーが「このコードベースを理解させて」と言った場合
- プロジェクト用のCLAUDE.mdの生成を求められた場合
- ユーザーが「オンボーディングして」「このリポジトリを案内して」と言った場合

## 仕組み

### フェーズ1：偵察

すべてのファイルを読まずにプロジェクトの生のシグナルを収集する。以下のチェックを並列実行：

```
1. パッケージマニフェスト検出
   → package.json, go.mod, Cargo.toml, pyproject.toml, pom.xml, build.gradle,
     Gemfile, composer.json, mix.exs, pubspec.yaml

2. フレームワークフィンガープリンティング
   → next.config.*, nuxt.config.*, angular.json, vite.config.*,
     django settings, flask app factory, fastapi main, rails config

3. エントリポイント特定
   → main.*, index.*, app.*, server.*, cmd/, src/main/

4. ディレクトリ構造スナップショット
   → ディレクトリツリーの上位2レベル（node_modules, vendor,
     .git, dist, build, __pycache__, .nextは除外）

5. 設定とツール検出
   → .eslintrc*, .prettierrc*, tsconfig.json, Makefile, Dockerfile,
     docker-compose*, .github/workflows/, .env.example, CI configs

6. テスト構造検出
   → tests/, test/, __tests__/, *_test.go, *.spec.ts, *.test.js,
     pytest.ini, jest.config.*, vitest.config.*
```

### フェーズ2：アーキテクチャマッピング

偵察データから以下を特定：

**技術スタック** -- 言語、フレームワーク、データベース、ビルドツール、CI/CD

**アーキテクチャパターン** -- モノリス、モノリポ、マイクロサービス、またはサーバーレス

**主要ディレクトリ** -- トップレベルディレクトリの目的マッピング

**データフロー** -- エントリからレスポンスまでのリクエストのトレース

### フェーズ3：規約検出

コードベースが既に従っているパターンを特定：命名規約、コードパターン、Git規約

### フェーズ4：オンボーディング成果物の生成

2つの出力を生成：

#### 出力1：オンボーディングガイド

概要、技術スタック、アーキテクチャ、主要エントリポイント、ディレクトリマップ、リクエストライフサイクル、規約、一般的なタスクを含む構造化されたMarkdown。

#### 出力2：スターターCLAUDE.md

検出された規約に基づくプロジェクト固有のCLAUDE.mdを生成または更新する。`CLAUDE.md`が既に存在する場合、まず読んで強化する -- 既存のプロジェクト固有の指示を保持し、追加・変更された内容を明確に示す。

## ベストプラクティス

1. **すべてを読まない** -- 偵察にはGlobとGrepを使い、すべてのファイルにReadを使わない。曖昧なシグナルにのみ選択的にReadを使用。
2. **推測せず検証する** -- 設定からフレームワークが検出されても、実際のコードが異なるものを使っている場合はコードを信頼する。
3. **既存CLAUDE.mdを尊重する** -- 既に存在する場合、置換ではなく強化する。
4. **簡潔に保つ** -- オンボーディングガイドは2分でスキャンできるべき。
5. **不明点はフラグを立てる** -- 確信を持って検出できない規約は、推測するより「テストランナーを特定できませんでした」の方が良い。

## 避けるべきアンチパターン

- 100行を超えるCLAUDE.mdの生成 -- 焦点を絞る
- すべての依存関係の列挙 -- コードの書き方に影響するものだけをハイライト
- 明白なディレクトリ名の説明 -- `src/`に説明は不要
- READMEのコピー -- オンボーディングガイドはREADMEにない構造的な洞察を追加する
