# Ralphinho RFC パイプライン

[humanplane](https://github.com/humanplane) スタイルのRFC分解パターンとマルチユニットオーケストレーションワークフローに着想を得たもの。

単一のエージェントパスでは大きすぎる機能を、独立して検証可能な作業ユニットに分割する必要がある場合にこのスキルを使用する。

## パイプラインステージ

1. RFCの取り込み
2. DAG分解
3. ユニットの割り当て
4. ユニットの実装
5. ユニットの検証
6. マージキューと統合
7. 最終システム検証

## ユニット仕様テンプレート

各作業ユニットには以下を含める：
- `id`
- `depends_on`
- `scope`
- `acceptance_tests`
- `risk_level`
- `rollback_plan`

## 複雑度ティア

- Tier 1: 単一ファイルの編集、決定論的テスト
- Tier 2: 複数ファイルにまたがる振る舞いの変更、中程度の統合リスク
- Tier 3: スキーマ/認証/パフォーマンス/セキュリティの変更

## ユニットごとの品質パイプライン

1. 調査
2. 実装計画
3. 実装
4. テスト
5. レビュー
6. マージ準備完了レポート

## マージキュールール

- 未解決の依存関係エラーがあるユニットは絶対にマージしない。
- ユニットブランチは常に最新の統合ブランチにリベースする。
- キューに入ったマージのたびに統合テストを再実行する。

## リカバリ

ユニットが停滞した場合：
- アクティブキューから除外する
- 調査結果のスナップショットを取る
- ユニットスコープを狭めて再生成する
- 更新された制約で再試行する

## 出力

- RFC実行ログ
- ユニットスコアカード
- 依存関係グラフのスナップショット
- 統合リスクサマリー
